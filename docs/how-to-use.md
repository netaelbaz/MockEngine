# How to Use

This page covers the real day-to-day workflows you'll use in the MockEngine portal.

## Create a Rule

1. Open the MockEngine web portal and go to **Rules** in the sidebar
2. Click **New Rule**
3. Fill in the rule form:

| Field | What to enter |
|---|---|
| **Name** | A unique, human-readable label (e.g. `Mock Login - 200`) |
| **URL Pattern** | The endpoint path to intercept. Three formats are supported: **exact** (`/api/users`), **wildcard** (`/api/users/*`), or **regex** (`^/api/users/\d+$`). Without a `*` or `^` the match is exact — `/api/users` will **not** match `/api/users/123`. |
| **Method** | The HTTP method (`GET`, `POST`, `PUT`, `DELETE`, `PATCH`, or `ANY`) |
| **Status Code** | The HTTP status code to return (e.g. `200`, `404`, `500`) — optional, not set by default |
| **Delay (seconds)** | Optional — adds an artificial delay before returning the response |
| **Mock Data** | The JSON body to return. Can be typed, pasted, or generated by AI — optional, empty by default |
| **Backend Mode** | See [Backend Mode](#backend-mode) below |

4. Click **Save**. The rule is created in an **enabled** state by default and takes effect on connected devices within one polling cycle.

> Rules are fetched by devices every 60 seconds. A newly enabled rule will take effect within one polling cycle.

## Backend Mode

Every rule has a **Backend Mode** that controls whether the real server is called:

### Mock Backend

The SDK returns the mock response directly. The real server is **never called**.

Use this when:
- The endpoint doesn't exist yet
- You don't care about collecting statistics from real server on this endpoint
- You're working offline

### Call Real Backend

The SDK makes the real HTTP request. After receiving the response, it replaces the status code and/or body with the rule's configured values.

Use this when:
- You need the server-side effects to happen (auth, writes, etc.)
- You want to override only part of the response
- You're testing how the app handles an unexpected response from a normally-healthy endpoint
- You want to collect statistics from real server on this endpoint

> The portal automatically suggests **Real Server** so statistics will always get collected on server activity. You can override this.

## Enable a Rule

1. Go to **Rules** in the portal
2. Find the rule in the list (use the search bar to filter by name or endpoint)
3. Click the **toggle** in the Enabled column to turn it on

The rule becomes active on all connected devices within 60 seconds.

## Disable a Rule

1. Go to **Rules** in the portal
2. Find the rule
3. Click the **toggle** in the Enabled column to turn it off

Matched requests will once again reach the real server.

## Editing a Rule

1. Go to **Rules** in the portal
2. Find the rule
3. Click the **editing icon** in the actions column to start editing.
4. Click 'Save Changes'


## Common Use Cases

### Mock a Response

To return a specific response body and status code:

1. Create (or edit) a rule
2. Set **Status Code** to the code you want (e.g. `200`)
3. Set **Mock Data** to a valid JSON object:

```json
{
  "id": 42,
  "username": "testuser",
  "email": "test@example.com",
  "token": "eyJhbGciOiJIUzI1NiJ9..."
}
```

4. Set **Backend Mode** to **Mock Backend** — the SDK will return this response immediately without calling the real server

> **Tip:** Use the **Generate with AI** button to describe the response in plain English and have the JSON generated for you.

### Simulate Latency

To add an artificial delay to a specific endpoint (useful for testing loading states, timeouts, and UI spinners):

1. Create or edit a rule targeting the endpoint
2. Set **Delay (seconds)** to the desired wait time (e.g. `3` for a 3-second delay)
3. The delay applies before the response is returned, regardless of backend mode

> For timeout testing, set the delay higher than your OkHttp `readTimeout`. The interceptor will sleep for the configured duration and then return the response.

### Replace a Payload

To intercept a real server call but replace (or enrich) its response body:

1. Create a rule targeting the endpoint
2. Do not touch **Status Code** if you want to use real server status code
3. Set **Mock Data** to the replacement JSON body in 3 different ways:
  - upload a JSON file
  - generate using AI
  - type manually
4. Set **Backend Mode** to **Call Real Backend** — the SDK will make the real request but replace the response body with your mock data

This is useful when you want the real server's side effects (database writes, auth) to happen but need to control what the app receives back.

### Simulate Server Down

To test how your app handles a server outage or error:

1. Create a rule targeting the endpoint (or use `ANY` method with a broad URL pattern to cover all endpoints)
2. Set **Status Code** to desired status code (e.g 500)
3. Leave **Mock Data** empty or set it to `{"error": "Service unavailable"}` ( again you can ask AI to genearte a json response for you )
4. **Backend Mode** can be both. Doesn't matter.

Your app will receive a 500 response immediately, letting you verify your error-handling UI, retry logic, and user-facing error messages.

### AI-Powered Mock Generation

If you need a realistic JSON payload but don't want to write it by hand:

1. In the rule form, click **Generate with AI**
2. Describe the response in plain English:
   - *"A user profile object with id, name, email, avatar url, and a list of 3 recent orders"*
   - *"A 429 rate limit error response with a retryAfter field in seconds"*
3. Click **Generate**
4. The generated JSON appears in the Mock Data field — review and edit as needed
5. Save the rule
